# AI Integration Task - Plan

## Assignment Summary
Build an automated pipeline that reads a client update CSV and loads it into a
destination system, replacing the current manual daily copy process. Handle data
quality issues in the sample file. Write a short note (max half page) explaining
how it works, how to keep data accurate/secure long term, and what can go wrong.

## Constraints
- Open source tools only (no paid services)
- All 8 data issues must be handled, logged, and explained
- No em dashes in any written output
- PDF for the short note

## Destination System
SQLite (built into Python, no installation required, zero cost, widely readable).
The database file (crm.db) can be opened with DB Browser for SQLite (free).

## Resolution for Each Issue

1. **DUPLICATE_ROW**: Group by client_id after date normalization. Keep the row
   with the most recent last_updated. Log both rows, state which was discarded
   and why.

2. **MISSING_PHONE**: Store NULL in the database. Log a WARNING with the client_id
   so it can be followed up. Do not reject the row.

3. **PORTFOLIO_FORMAT**: Strip commas from the value before parsing as a number.
   Store as numeric (2100000). Log the transformation.

4. **DATE_FORMAT**: Try parsing as YYYY-MM-DD first, then DD/MM/YYYY. Normalize
   to YYYY-MM-DD. Log the original value and the normalized result.

5. **INVALID_EMAIL**: Validate with a regex requiring local@domain.tld format.
   Store the value as-is but set email_valid=0. Log with clear message. Do not
   reject the row.

6. **MISSING_PORTFOLIO**: Store NULL. Log a WARNING. Do not reject the row.

7. **NEGATIVE_PORTFOLIO**: Store the value but set portfolio_flagged=1. Log the
   value and note that all other portfolio values in the file range from 640,000
   to 2,100,000, making -50,000 a clear outlier (likely sign flip or wrong field).

8. **PHONE_FORMAT**: Normalize all UAE numbers to +971 XX XXX XXXX format by
   stripping spaces/dashes, validating country code (+971), splitting into prefix
   (2 digits) and subscriber (7 digits), then reformatting. Store both the raw
   original and the normalized version.

## Database Schema

### clients
- client_id (INTEGER PRIMARY KEY)
- first_name, last_name (TEXT NOT NULL)
- email (TEXT), email_valid (INTEGER 0/1)
- phone_raw (TEXT) -- original as it appeared in the file
- phone (TEXT) -- normalized format
- portfolio_value (REAL) -- NULL if missing or unparseable
- portfolio_flagged (INTEGER 0/1) -- 1 if negative or suspicious
- last_updated (TEXT YYYY-MM-DD)
- data_issues (TEXT) -- JSON array of issue objects
- imported_at (TEXT) -- timestamp of first import
- updated_at (TEXT) -- timestamp of last update

### sync_runs
- id (PK AUTOINCREMENT)
- run_at, source_file
- rows_read, rows_inserted, rows_updated, rows_skipped, issues_found
- summary (TEXT)

### sync_issues
- id (PK AUTOINCREMENT)
- sync_run_id (FK)
- client_id, row_number
- issue_code, issue_detail, resolution
